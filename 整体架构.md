# 个性化旅游系统 — 整体架构

## 一、技术选型与设计理念

### 1.1 技术栈总览

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| **后端框架** | Flask 2.x | 轻量级、Blueprint 模块化路由、Jinja2 模板引擎 |
| **前端框架** | Vue 3 (CDN) | Composition API、响应式数据绑定、无需构建工具 |
| **ORM 框架** | SQLAlchemy 2.x / Flask-SQLAlchemy | Python 标准 ORM，支持 MySQL 连接池、事务管理、迁移 |
| **数据库** | MySQL 8.x | 关系型数据库，支持事务 ACID，适合结构化用户+日记数据 |
| **地图服务** | 高德地图 JS API 2.0 | POI 搜索 / 驾车-步行-骑行路线规划 / 地理编码 |
| **密码安全** | SHA256 哈希（可升级 bcrypt） | 单向加密，密码不可逆 |
| **图标库** | Font Awesome 6.4 | 统一的 UI 图标体系 |

### 1.2 核心设计决策

- **前后端混合渲染**：Flask Jinja2 渲染初始 HTML 页面 → Vue 3 在浏览器中接管交互逻辑
- **双模式 API 兼容**：后端每个接口同时支持传统表单提交和 JSON 请求，平滑迁移
- **Vue 自定义定界符 `{$ $}`**：避免与 Jinja2 的 `{{ }}` 插值语法冲突
- **高德地图对象非响应式**：AMap 实例用普通变量管理，避免 Vue 不必要的追踪开销
- **ORM 数据映射**：用 SQLAlchemy Model 替代字典操作，类型安全 + 自动建表 + 迁移管理
- **连接池管理**：通过 `SQLALCHEMY_ENGINE_OPTIONS` 配置连接池，避免频繁创建/销毁连接

### 1.3 关于 Redis 缓存的决策

> **结论：当前阶段暂不引入 Redis，但架构预留扩展接口**

| 维度 | 分析 |
|------|------|
| **数据规模** | 用户数 + 日记数据量小，MySQL 单表查询毫秒级响应，无性能瓶颈 |
| **推荐数据** | 景点/美食/名校数据为静态硬编码，每次请求直接返回，无需缓存 |
| **Session 管理** | Flask 默认使用客户端 Cookie 存储 session（签名加密），不依赖服务端存储 |
| **并发场景** | Demo 级别应用，单进程即可满足需求 |
| **何时引入** | 当出现以下场景时考虑加入 Redis：<br>① 用户量破万，登录状态查询成为热点<br>② 推荐算法变复杂（协同过滤/实时计算），结果需要缓存<br>③ 需要分布式部署（多实例共享 session）<br>④ 需要限流/排行榜/消息队列等高级功能 |

### 3.1 用户认证流程

```
用户访问受保护页面（/recommend, /diary, /map）
        │
        ▼
  后端 @login_required 装饰器检查 Session
        │
   ┌────┴────┐
   │ 有 session│          无 session
   ▼         │          ▼
 正常渲染页面 │     重定向到 /login
   (或返回JSON) │
```

**前端增强**：未登录时点击导航链接不再直接跳转，而是弹出「请先登录」提示弹窗：

- 点击「取消」→ 关闭弹窗，留在当前页
- 点击「去登录」→ 记住目标 URL → 跳转登录页 → 登录成功后可回跳

### 3.2 五大功能模块

| 模块 | 入口路径 | 核心功能 | 技术亮点 |

### 4.2 DDL 建表语句参考

```sql
-- 用户表
CREATE TABLE users (
    id          INT PRIMARY KEY AUTO_INCREMENT,
    username    VARCHAR(50)  NOT NULL UNIQUE COMMENT '用户名',
    email       VARCHAR(100) NOT NULL UNIQUE COMMENT '邮箱',
    password_hash CHAR(64) NOT NULL COMMENT 'SHA256 密码哈希',
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 日记表
CREATE TABLE diaries (
    id          INT PRIMARY KEY AUTO_INCREMENT,
    user_id     INT NOT NULL COMMENT '所属用户ID',
    title       VARCHAR(200) NOT NULL COMMENT '日记标题',
    content     TEXT COMMENT '日记内容',
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
CREATE INDEX idx_diaries_user_id ON diaries(user_id);
```

### 4.3 ORM 模型代码示例

```python
# models/user_model.py
from db import db

class User(db.Model):
    __tablename__ = 'users'

    id = db.Column(db.Integer, primary_key=True, autoincrement=True)
    username = db.Column(db.String(50), unique=True, nullable=False)
    email = db.Column(db.String(100), unique=True, nullable=False)
    password_hash = db.Column(db.String(64), nullable=False)
    created_at = db.Column(db.DateTime, default=datetime.utcnow)
    updated_at = db.Column(db.DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)

    # 关系：一个用户拥有多篇日记
    diaries = db.relationship('Diary', backref='user', lazy='dynamic', cascade='all, delete-orphan')

# models/diary_model.py
class Diary(db.Model):
    __tablename__ = 'diaries'

    id = db.Column(db.Integer, primary_key=True, autoincrement=True)
    user_id = db.Column(db.Integer, db.ForeignKey('users.id'), nullable=False)
    title = db.Column(db.String(200), nullable=False)
    content = db.Column(db.Text)
    created_at = db.Column(db.DateTime, default=datetime.utcnow)
    updated_at = db.Column(db.DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
```

### 4.4 Pickle → MySQL 迁移对照表

| 维度 | Pickle 方案 (旧) | MySQL 方案 (新) |
|------|------------------|-----------------|
| **存储介质** | `.pkl` 二进制文件 | MySQL InnoDB 引擎 |
| **唯一 ID** | `_next_ids` 字典手动自增 | `AUTO_INCREMENT` 主键自动递增 |
| **查询方式** | 内存 dict 遍历过滤 | SQL WHERE 条件 + 索引加速 |
| **并发安全** | 无（文件锁未实现） | 行锁 / MVCC 事务隔离 |
| **数据关系** | 嵌套字典模拟关联 | 外键约束 + ORM relationship |
| **导入导出** | 直接序列化整个 dict | SELECT 导出 / INSERT 批量写入 |
| **备份恢复** | 复制 pkl 文件 | mysqldump / binlog |
| **容量限制** | 受限于内存大小 | 磁盘空间，支持百万级记录 |

## 六、关键代码设计模式

### 6.1 单例服务模式（保持不变）

所有 Service 在模块级别实例化，整个应用共享一个实例：

```python
# services/user_service.py
class UserService:
    def __init__(self):
        self.db = db  # 注入 SQLAlchemy 实例

user_service = UserService()  # 模块级单例

# backend/routes/auth.py
from services.user_service import user_service  # 直接引用
user_service.verify_user(username, password)
```

**变化点**：Service 内部从 `DataStorage.load_data()` 改为 `db.session.query(Model)` 操作 ORM 对象。

### 6.2 Flask Blueprint 模块化（不变）

```python
# backend/app.py - 注册中心
def register_blueprints(app):
    app.register_blueprint(main_bp)       # /
    app.register_blueprint(auth_bp)       # /login, /register
    app.register_blueprint(diary_bp)      # /api/diary (url_prefix='/api')
    app.register_blueprint(recommend_bp)  # /api/spots, /api/food
    app.register_blueprint(rating_bp)     # /api/rating
```

### 6.3 数据库初始化模式（新增）

```python
# db.py - SQLAlchemy 引擎 + 连接池配置
from flask_sqlalchemy import SQLAlchemy

db = SQLAlchemy()

def init_db(app):
    db.init_app(app)

    with app.app_context():
        db.create_all()  # 自动根据 Model 创建所有表
        print("✅ 数据库表已就绪")
```

```python
# backend/app.py - 应用工厂中初始化
def create_app():
    app = Flask(__name__)
    app.config.from_object(Config)  # 包含 SQLALCHEMY_DATABASE_URI 等
    init_db(app)                    # 初始化数据库
    register_blueprints(app)        # 注册蓝图
    return app
```
